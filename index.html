🔄 GraphQL Subscriptions & Real-Time Data Streams

Tech Insight: Move beyond polling! GraphQL subscriptions leverage WebSockets to push real-time updates from server to client — think live chat, collaborative editing, stock tickers, and IoT dashboards. Unlike REST's request-response model or even periodic polling (wasteful & delayed), subscriptions establish a persistent bidirectional channel where the server pushes data exactly when it changes.

The architecture: Your GraphQL server subscribes to data sources (Postgres LISTEN/NOTIFY, Redis PubSub, Kafka topics) and relays events through WebSocket connections. Each client maintains a subscription query, and the server uses event-driven backends + connection batching to deliver updates with millisecond latency — even at scale.

# 🚫 OLD WAY: Polling every 3 seconds (wasteful, delayed)
const pollForUpdates = async () => {
  while (true) {
    const data = await fetch('/api/chat-messages'); // HTTP request
    renderMessages(data);
    await sleep(3000); // ⏱️ Max 3-second delay! Missed real-time.
  }
};
// ❌ Issues: Server load from empty polls, stale UI state,
//     bandwidth waste when nothing changes.

# ✅ NEW WAY: GraphQL Subscription (real-time push)
import { useSubscription } from '@apollo/client';
import { gql } from '@apollo/client';

const NEW_MESSAGE_SUB = gql`
  subscription OnNewMessage($chatId: ID!) {
    messageAdded(chatId: $chatId) {
      id
      sender { name avatar }
      content
      timestamp
      reactions { emoji userId }
    }
  }
`;

function ChatWindow({ chatId }) {
  const { data } = useSubscription(NEW_MESSAGE_SUB, {
    variables: { chatId },
    onSubscribe: (data) => updateUI(data), // ⚡ Instant!
  });
}
// ✅ Benefits: Zero polling overhead, sub-100ms latency,
//             scales efficiently — one connection per client.

# 🔧 Server-side (Node.js + Apollo Server + Postgres):
import { PubSub } from 'graphql-subscriptions'; // Or Redis for cluster
class ChatServer {
  async messageAdded(chatId, payload) {
    await db.query('INSERT INTO messages ...');
    pubPub.publish(`CHAT_${chatId}_MESSAGE`, { messageAdded: payload });
    // WebSocket clients auto-receive update via subscription!
  }
}

# 🌐 Production-scale with Redis PubSub:
const redis = new IORedis();
pubsub.subscribe(redis, ['NEW_MESSAGE']);
apolloServer.listen({ port: 4000 }, () => {
  // WebSocket upgrades to /graphql endpoint automatically.
});

Pro Tip: For massive scale (10k+ concurrent subscriptions), use a distributed pubsub backend like Redis Cluster or NATS, not in-memory GraphQL PubSub. Add heartbeat/ping-pong logic per WebSocket spec to handle dead connections auto-cleanup. Combine with Apollo Federation for microservices — each service publishes domain-specific events and federation gateway aggregates subscription streams into unified client queries!
