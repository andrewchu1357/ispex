🗄️ B+ Tree Indexes & Database Performance

Tech Insight: Every time you write a query with a WHERE clause, you're betting on the database's index structure. Most databases (PostgreSQL, MySQL, MongoDB) use B+ Trees for indexing — not because they're clever, but because they match how disk I/O actually works! Unlike binary trees that can be deeply nested and require many sequential reads from disk, B+ trees are fat: each node holds hundreds of keys at once, keeping tree height to just 3-4 levels even for billions of rows. Result? Finding any row requires reading only 3-4 random pages from disk.

The hidden reality: Disk seeks cost ~10ms (mechanical HDDs) or ~100μs (SSDs). Memory access is ~100ns. That's a 10,000x difference! A B+ tree with fanout of 100 needs only log₁₀₀(1 billion) ≈ 4.5 page reads to find any row. Even with SSDs eliminating mechanical seek time, I/O amplification from inefficient index structures still dominates query performance.

# 🌳 B+ TREE VS OTHER STRUCTURES:

# ❌ Binary Search Tree (too deep for disks):
Height = log₂(1 billion) ≈ 30 levels
→ Could need ~30 random I/O operations → 30×10ms = 300ms per query!
Every node holds: [left_ptr, key, right_ptr] — tiny nodes = many reads

# ✅ B+ Tree (optimized for page-aligned storage):
Height = log₁₀₂₄(1 billion) ≈ 3.2 levels  
→ Typically just 4 disk seeks → ~40ms total!
Each node holds: [ptr, key]×512 — fat nodes fit disk pages perfectly
Leaf nodes chain together for efficient range scans 🔄

How B+ Trees Work: Each internal node is exactly one disk page (usually 8KB-32KB). If each "key + pointer" pair takes up ~50 bytes, a single 16KB page can hold ~327 keys and pointers! This high fanout dramatically reduces tree height. Unlike binary trees where data sits in internal nodes, B+ trees store all actual records only in leaf nodes, while internal nodes act purely as directory layers pointing to ranges of values.

# 📊 REAL-WORLD EXAMPLE (PostgreSQL 8KB pages):

# Creating an index:
CREATE INDEX idx_users_email ON users(email);

# The EXPLAIN ANALYZE reveals B+ tree magic:
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'test@example.com';
└─ Index Scan using idx_users_email on users (cost=0.42..8.44 rows=1)
   → Actually executed: 3 page reads! ✅
   ┌── Root node in memory (hot cache, never hits disk)
   ├── Internal node: Find 'test@...' falls between 't' and 'u'
   │   └─ Pointer to leaf page #2847
   └── Leaf page contains actual row(s) + pointers to next/prev leaves
       (for range queries like BETWEEN or LIKE 'test%')

Why Not Just Use Hash Indexes? Hash indexes give O(1) lookup time for exact matches — faster than B+ trees! But they're useless for everything else: no range queries (BETWEEN, >=), no sorting (ORDER BY), no prefix searches (LICENSE LIKE 'abc%'). The chaining in leaf nodes also enables fast sequential scans — critical for "give me all users aged 25-34 sorted by signup date." B+ trees are the Swiss Army knife of indexing.

# 🎯 INDEX DESIGN STRATEGIES:

# ✅ COVERING INDEX (zero table lookups needed):
CREATE INDEX idx_orders_user_date ON orders(user_id, order_date);
SELECT user_id, total FROM orders WHERE user_id = 123 ORDER BY order_date;
→ All data in the index itself! No extra I/O to fetch rows.

# ❌ BAD: Index not used due to function on column:
SELECT * FROM users WHERE UPPER(email) = 'TEST@EXAMPLE.COM';
-- Function prevents B+ tree usage → full table scan!
FIX: Use expression indexes or normalize queries instead.

# ✅ COMPOSITE INDEX ORDER MATTERS:
CREATE INDEX idx ON (age, city);  -- Works for age AND "age + city" filters
SELECT * FROM users WHERE city = 'NYC';  ❌ Index NOT used! Leftmost rule applies.

Query Optimization Pro Tips: (1) Use EXPLAIN ANALYZE religiously — it shows actual vs estimated rows, index usage, and query plans. (2) Composite indexes follow the "leftmost prefix" rule: (a,b,c) works for queries on a, or a+b, but not just b. (3) Covering indexes include all columns you need in SELECT/JOIN/WHERE — zero extra lookups! (4) Partial indexes (WHERE is_active = true) save space and speed up filtered queries. (5) Watch out for "index scan" vs "bitmap index heap test" tradeoffs on large result sets.

# 🛠️ DIAGNOSTICS & TOOLS:

# PostgreSQL - See index usage stats:
SELECT schemaname, tablename, indexname, idx_scan, rows from pg_stat_user_indexes ORDER BY idx_scan ASC LIMIT 10;
-- Low idx_scan = unused indexes eating write performance!

# Show actual query plan with costs:
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) YOUR_QUERY_HERE;
→ Look for "Seq Scan" (full table scan — BAD if indexed)
→ Want: "Index Only Scan" or "Bitmap Heap Test"

# MongoDB - Query planner analysis:
db.users.find({age: {$gte: 25}}).explain("executionStats");
-- Check "executionStage": $IXSCAN = good, COLLSCAN = bad
